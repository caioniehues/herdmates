# Wave 8 learnings — direct socket backend (2026-07-15)

Retrospective for issue #8 and PR #38. Written before the release decision, per
the standing wave-close rule.

## Process learnings

### Persistent specialist panes beat disposable review rounds

Standards and Spec review found different classes of defects, but repeatedly
recreating reviewers discarded useful context and created pane churn. The
better team topology mirrors the development lifecycle: a persistent feature
implementer, Standards reviewer, Spec reviewer, and coordinator. Follow-up
rounds return to the same specialist; fresh reviewers are reserved for a real
need for independence or a second opinion.

### Adversarial review must follow the behavior into production callers

Early fixes added bounded reconnect helpers and tests, but the production
collectors did not use them. Later review followed Board and God wait call
paths and found repeated subscription creation, hot-spin fallback, reconnect
without a successful fresh snapshot, and terminal protocol errors treated as
transport loss. Review the production ownership path, not merely the helper
that appears to implement the requirement.

### RED-first review repair scales across rounds

Every actionable finding received a focused regression before repair. The
suite grew from 168 to 183 tests and locked down silent partial peers, total
deadlines, persistent multiplexing, typed validation, trace redaction,
snapshot-gated reconnect, terminal error handling, reconnect caps/backoff, and
shared lifecycle ownership. The coordinator repeated the complete suite 15
times after each delivered round.

### Live proof remains distinct from unit and review evidence

The final missing subscription trace was invisible to the green suite until a
reviewer ran the documented live probe and counted zero `events.subscribe`
rows. The post-merge proof compared CLI and socket verdicts on the same active
run and asserted exactly one redacted subscription row. Documentation commands
must be executable acceptance tests, not illustrative snippets.

## Design learnings

### Durable truth and live wakeups are different responsibilities

The socket backend supplies typed snapshot bootstrap and event wakeups, while
`run.toml` and inbox reports remain completion and verdict truth. Socket events
never manufacture worker completion. This separation preserves correctness
when the live stream disconnects, reconnects, or delivers malformed data.

### Long-lived subscriptions need an explicit lifecycle owner

One shared `CollectorSocketController` now owns snapshot gating, subscription
establishment, timeout preservation, reconnect counters, absolute deadlines,
backoff, and terminal/exhausted states. Board and God adapters only map typed
outcomes to UI refresh or remaining-budget CLI fallback. Centralizing the
state machine prevents policy drift between consumers.

### Error classification determines safe recovery

Ordinary read timeout preserves the stream. EOF and transport errors may
reconnect within cap/backoff/deadline bounds. Response-ID, framing, schema,
ACK, and typed-validation failures are terminal and fall back instead of
retrying. Every replacement subscription is gated on a successful fresh typed
snapshot.

### Observability must cover every protocol path

Tracing only request/response RPCs omitted long-lived subscription handshakes.
The final implementation emits one redacted outcome row for each
`events.subscribe` handshake, including early connect/write/frame/validation
failures, with fixed error categories and no server, prompt, or message text.
