# Wave 6 learnings — team wait + god CLI + god skill (2026-07-15)

Coordinator retrospective for #23+#24 (PR #30 + #33) and #25 (PR #31). Per
standing rule: formalize learnings at every task/wave close.

## Process learnings

### Live DoD demos catch what unit suites structurally cannot
PR #30 shipped with 149 tests, two clean review rounds, 0/15 flake loops —
and the live DoD immediately found two real bugs: (1) `workspace.closed`
reconciles a worker to `ended`, which the wait unsatisfiable-check didn't
count (only failed/orphaned), burning full timeout with a lying `timeout`
verdict; (2) newest-active-run auto-selection sorted lexicographically
(`wave6b` > `dod6`), picking an older run. Both classes are invisible to
in-process fakes because the fake's lifecycle vocabulary and run-dir naming
mirrored the author's assumptions. **Rule: DoD demos must drive the real
binary against real runs the coordinator perturbs (close workspaces, race
runs with contradicting name/timestamp order).**

### Verb-surface dependency between parallel workers: hold, don't guess
J wrote the skill against I's round-1 verb table; I's MAJOR fixes then
changed the surface (new exit 4, msg fan-out semantics). Holding J's
approved-but-unmerged PR until I's surface froze, then messaging J the exact
delta for a one-commit sync, kept both PRs honest with zero rework loops.
**Pattern: docs-worker merges last, syncs from the code-worker's final
REPORT delta, never from intermediate states.**

### A worker's own DoD list is a coordinator asset
I's REPORT carried "live re-verify for coordinator" steps; running them
verbatim (plus adversarial perturbations) is what surfaced the two live bugs.
Require this section in every brief's report format.

### Post-merge fixes go on a fresh branch off updated main
The `ended`-lifecycle + run-ordering fixes landed as PR #33 on
`fix/wait-terminal-lifecycles` cut from post-merge main — no rebase pain,
no reopening merged PRs, worker kept its context (same pane, new branch).

## Mechanics learnings

- **`msg` targets resolve against the newest run by default** — with
  multiple active runs, always pass `--run <dir>` when messaging a specific
  team (bit me: `msg godcli-i` resolved to J's run, listed J as candidate).
- **Pointer-without-report is a live, common state**: dod6's `silent` worker
  fired a completion pointer with no report; `inbox --unread` correctly
  showed `STOPPED-NOT-DONE`. The new verbs turn the old manual triage into
  one command — the design's redundancy (pointer + durable report) is load-
  bearing, not belt-and-braces.
- **Measured exit codes live**: 0 (any-report reached mid-run at 59s),
  1 (unknown worker, hard error with candidates), 2 (timeout), 3
  (dead_worker for `ended` worker, instant post-#33). Shell trap: piping a
  command through `head` then reading `$?` measures `head` — capture exit
  before piping.
- **env-mutating tests are a flake class**: paths.rs tests originally used
  `set_var`/`remove_var` under a module-private lock while the wider suite
  reads env on other threads — same class as the PR #27 flake. Fix pattern:
  extract a pure resolver taking values as parameters; zero env mutation in
  tests.
- **Compile-time manifest parity**: `include_str!("../herdr-plugin.toml")`
  in a test pins hardcoded ids to the manifest — drift fails the suite.

## Verified product wins

- `wait --until any-report` returned during a real run (59s, exit 0, one-line
  JSON verdict); bounded timeouts enforced end-to-end.
- `inbox --unread` + read-marks persist across invocations
  (`[hook.report_read_mtime_ms]`); re-written reports flip back to unread.
- Zero-ceremony invocation works env-less from a plain shell (XDG layout via
  manifest id, explicit env wins) — the ~90-char prefix is dead.
- `msg all`/comma fan-out prevalidates, filters terminal workers, aggregates
  per-target errors, and hard-errors on an empty live set.
- `skills/god/SKILL.md` shipped, verified against code (board keys, sentinel,
  exit codes) and synced to the final surface.
